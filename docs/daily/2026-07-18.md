## Task 1: frontend-ci を required 昇格できる形に（#697 / PR #698）

### 罠の正体（実測）

- **同名 job による context 衝突**: frontend-ci の job 名が `check`、backend-ci の job も `check`。GitHub Actions のチェック context は **job 名そのもの**になるため、`required = ["check"]` がどちらを指すか曖昧になりうる。
- **paths フィルタ付き required の永久 BLOCK**: frontend-ci には `paths: frontend/**` 等のフィルタがあり、backend-only PR では job が **skip** される。required に入れると status が付かず、**mergeable にならず永久ブロック**（vault/contact が実証済みの罠）。

### 対応（第三の道）

1. push / pull_request 双方から `paths:` フィルタを除去 → **全 PR/push で常時実行**。
2. job `check` → `frontend-check` にリネームし context を分離。
3. backend-ci の `check` は**不変**・`e2e` job は現状維持（required 対象外）。

### 検証

PR #698 上で 4 context 全緑を実測:

| context          | 由来                        | 結果       |
| ---------------- | --------------------------- | ---------- |
| `check`          | backend-ci                  | pass 48s   |
| `integration`    | backend-ci                  | pass 39s   |
| `frontend-check` | frontend-ci（リネーム反映） | pass 1m30s |
| `e2e`            | frontend-ci                 | pass 2m19s |

- **リネーム反映を確認**: context が `frontend-check` として立った（backend `check` と分離）。
- **paths 除去の実証**: この PR 自体が `frontend/` 無変更（`.github/` のみ）にもかかわらず frontend-ci が走った＝skip ブロックの罠を回避できている。

merge SHA = `089e8c3`。統合リナが branch protection の required を **`[check, frontend-check, integration]`** に PATCH（integration も昇格採用・e2e は flakiness 観測枠で保留）。

### 着手前の実測検証で TOP5 を 2 度訂正（この型が機能）

- **⑤ audit export の「formula-injection が audit だけ穴」は誤り**: フロントの export フックは server 生成 CSV を `getBlob` で DL するだけ＝injection 防御はバックエンド `Csv::putRow` 側。peer の export.test も injection でなくフック状態機械をテスト。→ ⑤の穴は「状態テスト欠落」で、**CSV injection 実テストはバックエンド案件（T3 候補）へ移送**。純ロジックの ① errors.ts を採用。
- **③ use-sign-in は zero test ではない**: `SignInForm.test.tsx`（UI テスト）が成功時トークン保存・不正時 alert＋非セッションを end-to-end で網羅済み。model テストは重複 → 真の zero test の ④ company-seal へ振り替え（「framework 挙動のテストは書かない」で hub と合意）。
