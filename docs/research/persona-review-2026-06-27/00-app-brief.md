# NeNe Invoice — ペルソナ評価用 製品ブリーフ（2026-06-27 実機ベース）

このドキュメントは、ペルソナ評価パネルが「製品の趣旨・機能を理解し、実機スクリーンショットを目視し、正常系シナリオを体験した」前提を作るための、実機（Docker, seed データ入り）に基づく正確な要約です。誇張せず、実際に確認できた挙動のみを記載しています。

## 1. これは何か（趣旨）

**自己ホスト型（セルフホスト）の見積・請求・入金管理 OSS**。日本の中小企業向けで、**適格請求書（インボイス制度）対応**。ライセンスは MIT（無料）。

- **最大の特徴＝自己ホスト**：SaaS（freee / マネーフォワード / Misoca / Board 等）と違い、**自社の借りているレンタルサーバ（ヘテムル・さくら・ロリポップ・エックスサーバー等の共用ホスティング）や Docker 環境に自分で設置**して使う。データは自社サーバ内に残る。月額課金なし。
- 設置方法：(A) 共用ホスティング向けに **ZIP をアップロード＋ブラウザのウェブインストーラ**で初期設定（本番 DB は MySQL）。(B) 開発・自社サーバ向けに Docker。
- マルチテナント設計（1 インスタンスで複数組織も可だが、通常は 1 社 = 1 組織）。UI は **日本語／English** 切替。

> 評価上の論点（重要）：対象ペルソナは「すでにレンタルサーバで自社サイトを持っている会社の会計決裁者」。つまり**サーバを借りている前提**はあるが、PHP アプリの設置・DB 作成・運用（バックアップ／アップデート）を自分（または出入りの制作会社）でやれるか、が採用の分かれ目になる。

## 2. 主要機能（実機で確認済み）

**ダッシュボード**：KPI カード（未払い請求書件数／期限超過件数／当月入金額・前月比／売掛残高合計）、当月請求発行額（着地見込み・前月比・前年同月比）、月別推移バーチャート（直近6ヶ月）、日別積み上がり折れ線、最近の未払い請求書、売掛金エイジング（期限内／1〜30日超過／31日以上超過）。

**見積書**：一覧（番号・取引先名で検索、状態フィルタ、有効期限・合計レンジ、全列ソート、CSV エクスポート）。作成（取引先検索、明細行＝品目・数量・単価・税率〔10%/8%〕・金額、明細追加、備考、小計／消費税／合計の自動計算、テンプレートから読込／テンプレートとして保存）。状態遷移（下書き→送付済み→承認済み／却下／期限切れ）。PDF 出力。

**請求書**：一覧（検索・状態・期限・金額フィルタ・ソート・CSV）。作成（見積書から変換、または直接作成）。発行（適格請求書フラグ＋発行者登録番号チェック、採番 INV-YYYY-NNN、支払期限）。詳細（明細、小計／消費税／合計／**残高**、入金履歴、入金記録フォーム、**PDF ダウンロード**、**メールで送信**、**クライアント向け決済リンクを生成**、この内容で複製）。状態（下書き／発行済み／一部入金／期限超過／支払済み）。

**入金**：請求書ごとに入金記録（金額・入金日・方法・備考）。一部入金で「一部入金」、満額で「支払済み」に自動遷移。残高表示。

**取引先**：一覧（名称・読み・担当・メール・登録番号で検索、ソート）、作成／編集／削除（論理削除）、買い手の適格請求書登録番号、CSV インポート。

**品目**：商品・サービスのカタログ（標準単価・税率）、CSV インポート／エクスポート。明細入力時に流用。

**テンプレート**：見積／請求の文書テンプレート（よく使う明細セットの保存・読込）。

**カード決済（PAY.JP）**：請求書ごとに公開決済ページのリンクを生成し、受領側がカード決済できる（hosted 型・カード情報は本アプリに保持しない＝SAQ-A）。webhook で入金反映。

**監査ログ**：全操作（作成・発行・送付・入金・DL トークン発行等）を記録。フィルタ・CSV。会計用語＋日英表示、操作者をメール表示。

**サービストークン**：外部システム連携用のトークン発行・一覧・失効（read:invoices / write:payments のスコープ）。例：入金消込・督促を行う別ツールと HTTP 連携。

**設定**：会社情報（発行者プロフィール、適格請求書登録番号、住所・電話・メール、銀行口座、締め日＋支払サイト〔支払条件デフォルト〕、PDF の体裁＝テンプレート／余白／見出しフォント、ロゴ・印影）。

**その他**：日本語／English 切替（localStorage 永続）、キーボードショートカット（一覧の j/k、n で新規、/ で検索、g 連打でナビ等）、レスポンシブ（モバイルはテーブルがカード化＋ボトムナビ）、ログインはメール＋パスワード（リロードでメモリ内トークンは消えるが httpOnly cookie でサイレント再認証）。

## 3. 体験してもらう正常系シナリオ（長め・一気通貫）

1. ログイン（admin@example.com）。
2. ダッシュボードで売掛状況・期限超過・エイジングを確認。
3. 取引先を 1 社新規登録（名称・読み・担当・メール・登録番号）。
4. 見積書を作成：取引先を検索選択、明細 2 行（例：コンサルティング5時間 @¥30,000／報告書作成 @¥50,000、税率 10%）、合計を確認、（必要なら）テンプレートとして保存。
5. 見積を PDF で確認 → 送付済みにし、先方承認後「承認済み」に。
6. 承認済み見積を**請求書に変換**。
7. 請求書を**発行**（適格請求書・登録番号あり）→ 採番（INV-2026-xxx）・支払期限設定。
8. 請求書 PDF をダウンロード／クライアントへメール送信／**クライアント向け決済リンクを生成**。
9. 後日、**入金を記録**（一部入金 → 残高が表示される → 残額入金で支払済み）。
10. ダッシュボード／監査ログで反映を確認、必要なら請求一覧を CSV エクスポート（会計ソフト取込用）。

## 5. 評価の観点（必ず触れること）

- 自分（その業種・立場・IT 習熟度）にとって**採用／検討／見送り**のどれか、なぜか。
- 「自己ホスト（自分のレンタルサーバに設置）」を**現実的にやれる／やれない**か、誰がやるか。
- 既存手段（Excel・手書き・freee/MF 等 SaaS・税理士丸投げ）と比べた損得。
- **適格請求書・インボイス対応**は信用できるか。
- **何が良かった／ダメだった／分かりにくい／要らない／欲しい**。
- **いくらなら／どういう条件なら、無料でなく「金を払ってでも」使いたいか**（保守・サポート・設置代行・クラウド版など）。
