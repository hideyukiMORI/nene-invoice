# 統合レポート（アナリスト）

## 1. エグゼクティブサマリ

製品の「画面・インボイス対応・見積→請求→入金の一気通貫」は10人中ほぼ全員が高評価したが、**採用0／検討2／見送り8**。検討の2人(佐藤・高橋)はいずれも自分でVPSを建てられるITリテラシ高層で、見送り8人の離脱理由は機能以前に「**自己ホストを自力で運用できない**」に集約される。支払意思ありは**3人(田辺・佐藤・渡辺)**だが、その3人ですら「無料の自己ホスト版そのものには1円も出さない/出しても使えない」と明言し、金を出す条件は全員が**マネージドなクラウド版**または**保守・設置代行**に紐づく。結論として、現状の「自己ホストOSS単体」は技術者ニッチには刺さるが、日本SMBの会計決裁者の大多数には構造的に届かない。

- 判定内訳: 採用 0 / 検討 2(佐藤・高橋) / 見送り 8
- 支払意思: あり 3 / なし 7
- 致命的分岐: 自己ホストを「やれる」2 vs「絶対無理」8

**P2. 定期請求(自動発行)が無い ― 実質8/10**

**P3. 入金消込が手入力・銀行連携なし ― 4-5/10**
渡辺・木村にとっては「判定の生死を分ける本丸」。渡辺「振込名義が物件名や旦那名でバラバラ、誰の入金か当ててくれない」、木村「信金から何百人分まとめて来るのに一枚ずつ手消し=地獄」。佐藤・岡田も「freeeの自動消込からは明確な退化」。

**P4. サポート窓口・保証がない(無保証・自己責任の明記) ― 7-8/10**
田辺・中村・岡田・森・渡辺・木村・清水・李。中村「インボイスに不安な人間に『無保証・自己責任』は一番怖い」。ログイン画面の「税務・会計の最終確認は利用者の責任」が逆効果に働いている。

佐藤「会計は別途必要でfreeeを切れない=月額の痛みは半分」、森「MFの代わりにならず管理が逆に増える」。田辺・中村・渡辺・木村は「税理士の弥生/freee/MFに確実に取り込めるCSVか不明」を不安視。

**P6. 業種固有の必須項目の欠如 ― 各1-3名(が致命傷)**

## 5. 自己ホストの壁(温度感の分布)

**設置・運用が「障壁にならない/むしろ採用理由」: 2/10**
佐藤「VPSもXserverも持っている、設置は壁にならない」、高橋「docker composeで数分、cronでgit pull＋DBダンプで済む」。両者ともバックアップ・アップデートを「自動化で回せる、不安ゼロ」と明言し、**ここに課金されると自己ホストの意味が消える**と主張(高橋・佐藤)。

**「絶対無理・やる人もいない」: 8/10**
全見送り組。田辺・李・渡辺・木村はロリポップ/共用サーバにHPを置くだけのリテラシで「PHP設置・DB作成は外国語」。中村「現場の息子に一生やらせるのは無理」、清水・森・岡田「3人/2人/一人の事務所が財務・顧客データを守りきれない」。

**派生する不安(複数)**:
- 7年保存・電子帳簿保存法・バックアップの責任が全部自社に来る: 田辺・岡田・森・渡辺・木村・清水
- PHP 8.4要件で共用ホスティングが動くか不明: 佐藤・岡田・森・渡辺
- 提供元保証がなく消費税端数処理・法改正に誰が追従するか不明: 岡田・森・木村

**所見**: 自己ホストは「2 vs 8」のきれいな分断であり、製品の中核思想がそのまま市場の8割を弾いている。これは機能追加では埋まらず、**配布モデル(提供形態)の問題**。

## 6. 価格・支払意思の分析

**無料のまま使う層 vs 有料化**
- 純粋に「無料の自己ホストで十分、1円も払わない」: 高橋(GitHub Sponsors月500-1,000は別)、佐藤(本体は¥0、中身が増えた時だけ)。
- 「無料でも使えないので価値ゼロ」: 李・中村・木村・清水・田辺。**タダかどうかが問題ではなく、運用できないことが問題**(李「タダでも価値ゼロ、金の問題じゃない」)。

**刺さる課金形態(=金が動く条件)**

| 形態 | 想定価格帯 | 支持者 |
| --- | --- | --- |
| マネージドなクラウド版(運用・保守・サポート・法改正追従込み) | **月1,000〜3,000円** | 田辺・李・岡田・森・木村・清水・渡辺(7人) |
| 設置代行(初期一括) | 1〜3万(李・木村)〜3〜5万(田辺・中村・岡田・清水)〜5〜10万(渡辺) | 6人 |
| 月額/年額の保守契約(更新・バックアップ・電話窓口) | 月1,000〜5,000 または 年1〜3万(佐藤) | 田辺・中村・渡辺・岡田・佐藤 |
| 機能解放の買い切りアドオン(多通貨+Stripe+定期請求) | 3,000〜5,000円(買い切り) | 高橋 |
| OSS存続支援(GitHub Sponsors) | 月500〜1,000円 | 高橋 |

**最重要の落とし穴(矛盾)**: クラウド版に月千〜三千円を払うと言った人のほぼ全員が、同じ口で「**その額を払うなら、会計連携も法改正追従もあるMisoca(月800円〜)/freee/MF/boardを選ぶ**」と述べた(岡田・森・木村・中村・渡辺・清水)。つまりクラウド化しても**既存SaaSに対する差別化(EC連携・自動消込・源泉徴収・PDF美麗さ等)が無ければ価格競争で負ける**。李は「月数件なら無料SaaSや税理士で足り、わざわざ払う動機が弱い」と、クラウド版が出ても需要が薄い少額利用者の本音も漏らした。

## 7. 「採用」に必要な条件 — 優先順位つき提言

**P0(これ無しに採用は1件も増えない)**
1. **定期請求(継続/月次の自動発行)**。検討中の佐藤が「入れば請求レイヤーを採用」と明言、岡田・渡辺・木村も第一級。自己ホスト派にも非エンジニア派にも共通の最大公約数で、コア(無料セルフホスト)に入れるべき唯一の機能。
2. **write:invoices API + webhook**。佐藤・高橋(検討2人)の自動化要件。技術者ビーチヘッドを採用に転換する鍵。read偏重スコープのままでは自動化好きの期待に届かない。
3. **seed/初見UXの修正**。¥0/-100%の故障見え・開発者用語のメニュー露出は実装コスト低・離脱インパクト大。8人が指摘した最も安い改善。

**P1(セグメントを開くために必要)**
4. **マネージドなクラウド版の提供**(7人が第一に要求)。ただし**無料セルフホストコアは死守し二階建てに**(高橋・佐藤・清水が合意)。混ぜるとデータ主権派が死ぬ。
5. **会計ソフト取込の保証**(freee/MF/弥生に確実に入るCSV)＋**自動入金消込/銀行連携**(渡辺・木村の生命線、岡田・佐藤の退化指摘)。
6. **サポート窓口・保証・法改正追従の約束**。無保証表記が逆効果(中村)。「責任の所在」が非エンジニア層の安心の前提。

**P2(特定業種を取りに行く)**
7. 源泉徴収欄(佐藤・清水)、見積の単位欄+工種内訳(中村)、納品書+月締め(田辺)、口座振替一括消込(木村・渡辺)、多通貨+英語PDF+Stripe(高橋)、領収書(木村)、PDFブランド作り込み(清水)。

## 8. 「金を払ってでも使いたい」に必要な条件 — 具体策

支払意思の構造から、**収益化の主戦場は「自己ホストの肩代わり」+「業種特化の差別化」**の二点に絞られる。

- **二階建てモデルの確立**: 無料セルフホストコア(定期請求込み)＋有料マネージドクラウド(月1,000〜3,000円)。これがターン6でほぼ全員が収束した結論。クラウドに「設置・更新・バックアップ・電話サポート・法改正追従・税理士OKの取込」を全部束ねれば、田辺・李・森が即客になる価格帯。
- **既存SaaSへの明確な優位を1つ作る**: クラウド版単体ではMisoca/freeeに負ける(6人が明言)。**勝ち筋は森が言う「この見た目＋EC連携」、清水の「美麗PDF＋源泉徴収」のように、incumbentに無い差別化を尖らせること**。価格を下げるだけでは負ける。
- **技術者には機能の買い切りアドオン**: 高橋が「多通貨+Stripe+定期請求の買い切り3,000〜5,000円なら即買う」。月額課金を嫌う自己ホスト派から収益を取る唯一の形。佐藤も「保守チャネルなら年1〜3万」。
- **設置代行+保守パッケージ**: 初期3〜5万＋月千〜五千(田辺・中村・渡辺)。ただし「設置代行を付けても手入力消込・手複製のままなら乗り換えない」(渡辺)ため、**P0/P1機能とセットでなければ売れない**。
- **大前提=税理士のお墨付き**: 田辺「税理士が『この保存で大丈夫』と言わなければ1円も出せない」。会計事務所向けの認定・チェック済みラベルが支払いの最終ゲート。

## 9. セグメント別の勝ち筋

**最優先ビーチヘッド: ITリテラシの高い小規模BtoB(受託開発・Web制作・技術寄りの一人〜数名事業者)**
唯一、自己ホストOSSが現状のまま機能する層。佐藤・高橋の2人だけが検討まで来ており、**定期請求＋write:invoices APIを足せば即採用**になると本人が明言。データ主権・無料・docker一発・キーボード操作・国内インボイス対応という製品の強みが全部刺さる唯一のセグメント。ここを最初に取り、OSSコミュニティ・口コミの起点にする。高橋の多通貨要件は買い切りアドオンで収益化。

**第二: BtoB継続請求の士業・制作系(行政書士/デザイン事務所)**

# 付録C — 手法・再現性メモ

- **ペルソナ設計**: 業種・役職・年齢・経歴・IT習熟度・現状の請求手段・利用ホスティング・重視点／警戒点を10通りに分散（製造／Web制作／飲食／建設／士業／EC／フリーランス開発／不動産／学習塾／デザイン）。
- **フェーズ構成**: 個別評価10並列 → 討論6ターン×10名=60 → 統合1（計71サブエージェント）。各個別評価エージェントは製品ブリーフと実機スクショを `Read` で視覚的に取得し、開発者ではない一般ユーザーとして辛口評価するよう指示。
- **実機**: `docker compose up -d --build`（API＋ビルド済みSPA＋seed入りMySQL）。`admin@example.com` でログインし Playwright で撮影。
- **限界**: ペルソナはLLMによる擬似的なユーザーであり、実ユーザー調査の代替ではない。スクショ目視ベースで、実際のクリック操作・速度・エラー遭遇は再現していない。価格帯・支払意思は示唆であり実購買の証拠ではない。
