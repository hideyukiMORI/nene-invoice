## 🔴 A. LP brief §6 の事実ガードレールが陳腐化している（要施主判断・コンプラ）

`docs/marketing/nene-invoice-lp-brief.md` §6 に:

> ❌ 「銀行と自動連携／自動消込」と書かない（**入金は手動記録**。自動突合は NeNe Clear 側）。

これは **#505（銀行入金 自動消込エピック）が入る前**の記述。**v1.0.0 は invoice 自身に
`/bank-reconciliation` を搭載**（CSV 取込〔Shift_JIS〕→ 名義ゆれ辞書照合 → スコアリング →
**確認起票**）。CHANGELOG [1.0.0] にも記載済み。→ ガードレールが「今や事実である機能」を
禁じてしまっている。

**ただし単純に反転させると今度は過大表現になる**ので、正確なニュアンスで直す必要がある
（会計プロダクト・非交渉）:

- ✅ 言ってよい: 「**銀行入金 CSV の取込＋名義照合＋確認して一括消込**（半自動・人が確認して確定）」
- ❌ まだ言ってはいけない:
  - 「**銀行と自動連携**」= ライブの銀行 API 接続は無い（invoice は CSV 取込）。
  - 「**全自動で消込**」= 消込は人の確認起票（confirm-to-record）。手数料差引 write-off・
    過入金按分は**税理士ゲートで未実装**（#543）。

**提案する §6 の書き換え（施主/統合リナ承認後に反映）**:

> - ❌ 「**銀行と（口座）自動連携**」「**全自動で消込**」とは書かない（ライブ銀行 API 接続は無く、
>   消込は CSV 取込＋名義照合＋**人が確認して起票**する半自動。手数料 write-off／過入金按分は
>   税理士ゲートで未搭載）。
> - ✅ 「**銀行入金 CSV の取込・名義照合・確認消込**（`/bank-reconciliation`）」は事実、訴求 OK。

> **✅ 反映済み（2026-07-17・施主 GO）**: 上記の精密ニュアンス版を `nene-invoice-lp-brief.md` §6 に反映した。
> LP brief は §6 込みで Claude Design 渡し可能な確定状態。
